# [기획서] LLMAPI 연동 Client 서버 구축안 (v0.2)

작성일: 2026-04-12  
문서 목적: LLMAPI를 호출하는 중간 Client 서버의 구축 목표, 범위, 운영 전략을 정의한다.

---

## 1. 프로젝트 개요

### 1.1 배경
업무 시스템(상담관리, QA, VOC 등)에서 LLMAPI를 직접 호출하면 시스템별 구현이 달라져 아래 문제가 발생한다.

- 요청 포맷이 제각각이라 유지보수가 어려움
- 재시도/타임아웃/에러처리를 각 시스템이 중복 구현
- 장애 시 원인 파악이 오래 걸림

### 1.2 목적
Client 서버를 두어 연동을 표준화하고, 요청 이력/분석 결과를 MySQL에 저장하여 운영 추적성과 감사 가능성을 높인다.

### 1.3 기대효과

- 연동 표준화로 개발 속도 향상
- 장애 대응 시간 단축(trace_id + DB 이력)
- 요청/결과 데이터 자산화(리포트/재처리/감사 대응)

---

## 2. 목표와 성공 기준

### 2.1 목표

1. 동기/비동기 분석 요청을 안정적으로 중계한다.
2. 요청 이력과 결과를 MySQL에 구조적으로 저장한다.
3. 상태 전이 이력을 남겨 운영자가 처리 흐름을 추적할 수 있게 한다.
4. 재시도/타임아웃/에러 표준화를 통해 운영 안정성을 확보한다.

### 2.2 KPI

- 요청 성공률(완료율)
- 동기 요청 p95 지연시간
- 비동기 큐 평균 대기시간
- 5xx 오류율
- 재시도 후 복구율

---

## 3. 시스템 경계 및 역할

### 3.1 Client 서버 역할

- 외부 업무 시스템 요청 수신
- 요청 검증 및 LLMAPI 형식으로 변환
- LLMAPI 호출(동기/비동기)
- MySQL 요청/결과/상태이력 저장
- 조회 API 제공(요청 상태, 결과, 오류)

### 3.2 LLMAPI 역할

- 실제 분석 처리(summary/sentiment/category)
- 내부 fail-over, circuit breaker, prompt 라우팅

### 3.3 업무 시스템 역할

- 분석 요청 생성
- 결과 조회 및 업무 반영
- PII 마스킹 규칙 준수

---

## 4. 범위 정의

### 4.1 In Scope (MVP)

- `/client/analyze` 동기 요청
- `/client/analyze/async` 비동기 등록
- `/client/analyze/async/{job_id}` 상태 조회
- MySQL 저장(요청, 결과, 상태이력)
- 표준 오류코드/메시지
- 기본 관측성(로그/헬스체크)

### 4.2 Out of Scope (후속)

- 관리자 웹 콘솔
- 실시간 대시보드 UI
- 다중 리전 재해복구

---

## 5. 운영 시나리오

### 시나리오 A: 동기 성공

1. 업무 시스템이 Client 서버에 요청
2. Client 서버가 요청 저장(`RECEIVED`)
3. LLMAPI 동기 호출
4. 결과 저장(`COMPLETED`) 후 응답

### 시나리오 B: 동기 실패

1. LLMAPI 응답 실패(502/timeout)
2. Client 서버 재시도 정책 수행
3. 실패 시 `FAILED` 저장 + 표준 오류 반환

### 시나리오 C: 비동기 처리

1. 업무 시스템이 비동기 요청
2. `QUEUED` 저장 후 `job_id` 반환
3. 워커 처리(`PROCESSING` -> `COMPLETED/FAILED`)
4. 조회 API로 상태 확인

---

## 6. 리스크 및 대응

### 6.1 주요 리스크

- LLMAPI 일시 장애
- 큐 적체
- 중복 요청
- 민감정보 로그 유출

### 6.2 대응 전략

- 재시도 + 타임아웃 + 에러 표준화
- 큐 모니터링 + 워커 수 확장
- 멱등키(`source_system + client_request_id`) 적용
- 마스킹 텍스트 저장 원칙

---

## 7. 개선 검토 후 추가 반영 항목 (v0.2)

초안(v0.1) 대비 아래 내용을 보강한다.

1. 데이터 거버넌스
- 보관기간 정책(TTL/아카이브/삭제)
- 조회 권한 범위(시스템/조직 단위)

2. 신뢰성
- 상태 전이 이력 테이블 필수화
- 요청 멱등성 정책 필수화

3. 운영성
- 장애코드 표준 분류(예: UPSTREAM_TIMEOUT, VALIDATION_ERROR)
- 재처리 API(실패건 재큐잉) 후속 설계 항목 추가

4. 보안
- 서버 간 인증(API Key 또는 mTLS) 요구사항 명시
- 감사 로그 필드(호출자, IP, 사용자/시스템 ID) 추가
