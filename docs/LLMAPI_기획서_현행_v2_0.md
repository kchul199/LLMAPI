# [기획서] LLMAPI 현행 운영 기획서 (v2.0)

작성일: 2026-04-12  
문서 목적: 현재 저장소의 실제 구현 상태를 기준으로, 서비스 목표/범위/운영전략을 명확히 정의한다.

---

## 1. 프로젝트 배경

콜센터 운영에서는 상담 품질 평가, VOC 분류, 감정 동향 파악이 중요하지만 수작업 분석은 다음 한계가 있다.

- 처리 시간이 길다.
- 평가자마다 기준이 달라 일관성이 낮다.
- 대량 트래픽에서 실시간 대응이 어렵다.

LLMAPI는 이 문제를 해결하기 위해 상담 텍스트를 표준화된 API로 받아 자동 분석하는 중간 계층(Orchestration API)으로 설계되었다.

---

## 2. 목표와 기대효과

### 2.1 목표

1. 상담 텍스트를 자동 분석해 요약/감정/카테고리를 빠르게 제공한다.
2. LLM 장애 상황에서도 서비스 가용성을 유지한다.
3. 운영자가 설정(프롬프트, 타임아웃 등)을 유연하게 조정할 수 있게 한다.
4. 신규 개발자도 빠르게 온보딩 가능한 단순한 운영 모델을 제공한다.

### 2.2 기대효과

- 상담 후처리 시간 단축
- QA 기준의 표준화
- 장애 전파 최소화(재시도/페일오버/회로차단)
- 비동기 큐 기반 확장 준비

---

## 3. 사용자 및 이해관계자

- 운영 담당자(콜센터 QA)
  - 분석 결과를 대시보드/업무시스템에 반영
- 백엔드 개발자
  - API 연동, 장애 대응, 성능 튜닝
- 플랫폼 운영자(인프라)
  - Docker/Redis/모니터링 운영

---

## 4. 범위 정의 (Scope)

### 4.1 In-Scope (현재 구현)

- 동기 분석 API
- 비동기 큐 제출/조회 API
- 헬스체크 API
- 입력 검증 및 응답 정규화
- Retry + Fail-over + Circuit Breaker
- Prompt version 라우팅
- Redis queue worker(옵션)
- CI 기본 게이트(컴파일/단위테스트)

### 4.2 Out-of-Scope (차기)

- DLQ(Dead Letter Queue)
- 멀티 테넌트 권한 모델
- 고급 모니터링 대시보드(메트릭 시각화)
- 정책 기반 PII 자동 검출/마스킹

## 5. 기능 요구사항

### FR-01. 동기 분석
- 클라이언트는 `/v1/analyze`로 분석을 요청할 수 있어야 한다.
- 요청 task는 `summary`, `sentiment`, `category`만 허용한다.

### FR-02. 비동기 분석
- 클라이언트는 `/v1/analyze/async`로 작업을 등록할 수 있어야 한다.
- 서버는 `job_id`를 반환하고, 상태 조회 API를 제공해야 한다.

### FR-03. 장애 허용
- 메인 LLM 호출 실패 시 백업 LLM으로 전환해야 한다.
- 연속 실패 시 Circuit Breaker를 통해 호출을 차단해야 한다.

### FR-04. 프롬프트 버전 관리
- 클라이언트는 `options.prompt_version`으로 특정 프롬프트 버전을 지정할 수 있어야 한다.
- 지정 버전이 없으면 기본 버전(v1)로 안전하게 동작해야 한다.

### FR-05. 운영 가시성
- `/v1/health`에서 아래 정보를 확인할 수 있어야 한다.
  - LLM 구성
  - Retry/Timeout/Circuit 상태
  - Prompt 파일 상태 및 버전
  - Queue 상태

---

## 6. 비기능 요구사항

### NFR-01. 가용성
- 업스트림 장애 시에도 가능한 한 요청 처리를 지속한다.

### NFR-02. 신뢰성
- 요청 단위 추적(`trace_id`)이 가능해야 한다.

### NFR-03. 확장성
- Redis 큐를 활성화하면 비동기 처리량 확장이 가능해야 한다.

### NFR-04. 운영성
- 환경변수로 주요 정책(재시도, 타임아웃, 큐)을 조정할 수 있어야 한다.

---

## 7. 운영 시나리오

### 시나리오 A: 정상 동기 분석
1. 상담 시스템이 `/v1/analyze` 요청
2. LLM 응답 성공
3. 결과 저장 및 UI 표시

### 시나리오 B: 메인 모델 장애
1. 메인 LLM timeout
2. 재시도 후 실패
3. 백업 LLM으로 페일오버
4. 결과 반환(`is_fallback=true`)

### 시나리오 C: 트래픽 급증
1. 비동기 API로 큐 적재
2. 워커가 순차 소비
3. 클라이언트는 `job_id`로 상태 확인

---

## 8. KPI 제안

- 평균 응답시간(동기) `latency_ms`
- 백업 전환율(실패 대비)
- 큐 대기열 길이(`queue_depth`)
- 실패율(502/500 비율)
